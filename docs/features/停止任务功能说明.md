# 停止任务功能说明

## 功能概述

实现了任务停止功能，允许用户在任务执行过程中主动停止正在运行的任务。停止任务后，系统会：
1. 终止求解器进程（SPEOS、FLUENT、Mechanical等）
2. 释放Redis分布式锁
3. 更新任务状态为CANCELLED
4. 允许排队的任务继续执行

## 实现的功能

### 1. Worker端改进

**文件**: `worker_windows/worker_app.py`

#### 进程管理
- 添加了 `_running_tasks` 全局字典，保存正在运行的任务进程信息
- 实现了 `register_running_task()`、`unregister_running_task()`、`get_running_task()` 函数
- 在执行任务时注册进程，任务完成时取消注册

#### 停止信号处理
- 添加了 `task_cancelled` 事件，用于检测任务取消信号
- 在 `_execute_solver()` 中定期检查取消标志
- 如果检测到取消信号，优雅终止求解器进程

#### 锁管理
- 任务开始时注册锁信息到 `_running_tasks`
- 支持在停止时释放锁

### 2. API端点

**文件**: `app/routers/tasks.py`

#### 停止任务API
- **端点**: `POST /api/tasks/{task_id}/stop`
- **功能**:
  1. 检查任务状态（只能停止正在运行的任务）
  2. 撤销Celery任务（`revoke(terminate=True)`）
  3. 查找并终止求解器子进程（通过进程名和命令行匹配）
  4. 释放Redis分布式锁
  5. 更新任务状态为CANCELLED

#### 支持的求解器
- SPEOS: `SPEOSCore.exe`, `speoshpc`
- FLUENT: `fluent`, `fluent.exe`
- Mechanical: `ansys`, `ansys.exe`
- Maxwell: `Maxwell`, `Maxwell.exe`
- CFX: `cfx5solve`, `cfx5solve.exe`

### 3. Schema定义

**文件**: `app/schemas.py`

添加了 `StopTaskResponse` 模型：
```python
class StopTaskResponse(BaseModel):
    task_id: str
    status: str  # CANCELLED
    message: str
    terminated_processes: List[Dict[str, Any]]
    lock_released: bool
    previous_status: Dict[str, str]
```

## 使用流程

### 用户操作流程

1. 用户在任务列表中看到正在运行的任务
2. 点击"停止"按钮
3. 系统显示确认对话框
4. 用户确认后，调用 `POST /api/tasks/{task_id}/stop`
5. 系统执行停止操作：
   - 终止Celery任务
   - 终止求解器进程
   - 释放Redis锁
   - 更新任务状态
6. 返回结果，UI更新任务状态为CANCELLED
7. 排队的任务可以开始执行

### 系统执行流程

```
用户请求停止任务
    ↓
API检查任务状态
    ↓
撤销Celery任务 (terminate=True)
    ↓
查找求解器进程（根据solver_type）
    ↓
终止进程（先terminate，5秒后kill）
    ↓
释放Redis锁
    ↓
更新任务状态为CANCELLED
    ↓
返回结果
```

## 技术细节

### 1. 进程终止机制

使用 `psutil` 库查找和终止进程：
- 根据求解器类型匹配进程名
- 通过命令行参数匹配任务（检查工作目录）
- 先尝试优雅终止（`terminate()`），等待5秒
- 如果进程未终止，强制杀死（`kill()`）

### 2. 锁释放机制

直接操作Redis删除锁：
- 锁的key: `solver:serial_execution_lock`
- 直接删除锁，不检查锁的拥有者（因为任务已停止）
- 确保锁被释放，让排队的任务可以继续

### 3. 状态管理

- 停止前状态：RUNNING, STARTED, PROGRESS等运行状态
- 停止后状态：CANCELLED
- 状态历史中记录停止操作，包括终止的进程信息

## 注意事项

### 1. 依赖项

确保安装了 `psutil`：
```bash
pip install psutil
```

如果未安装，需要在 `requirements.txt` 中添加：
```
psutil>=5.9.0
```

### 2. 权限要求

- 需要进程管理权限（通常管理员权限）
- Windows上可能需要以管理员身份运行

### 3. 进程查找限制

- 进程查找通过进程名和命令行参数匹配
- 如果多个任务使用相同的工作目录，可能误终止其他任务（但概率很低）
- 建议确保每个任务使用独立的工作目录

### 4. 锁释放

- 停止任务时直接删除Redis锁，不考虑锁的拥有者
- 这是安全的，因为任务已停止，不应该再持有锁
- 如果任务已经完成并释放了锁，删除操作会失败但不影响功能

### 1. 功能测试

### 2. 边界情况测试

- [ ] 停止已完成的任务（应该返回400错误）

- [ ] 多个任务排队时，停止第一个任务，第二个任务应该立即开始

## 与旧版本的区别

### 之前的问题

1. **无法停止正在运行的任务**
   - 用户发现错误时，无法停止任务
   - 只能等待任务完成或强制关闭worker进程

2. **强制关闭worker的问题**
   - 锁不会被释放
   - 导致后续任务无法执行
   - 任务状态不一致

### 现在的改进

1. **支持主动停止**
   - 用户可以通过API停止任务
   - 支持所有求解器类型

2. **优雅终止**
   - 先尝试优雅终止进程
   - 自动释放锁
   - 更新任务状态

3. **锁自动释放**
   - 停止任务时自动释放锁
   - 确保排队的任务可以继续执行

## 后续优化建议

1. **进程查找优化**
   - 可以考虑在worker中保存进程PID，避免进程查找的复杂性
   - 通过共享存储（Redis）传递进程信息

2. **WebSocket通知**
   - 停止任务后通过WebSocket通知前端
   - 实时更新任务状态

3. **权限控制**
   - 添加权限检查，只允许有权限的用户停止任务
   - 可以添加操作日志

4. **批量停止**
   - 支持批量停止多个任务
   - API: `POST /api/tasks/batch-stop`
